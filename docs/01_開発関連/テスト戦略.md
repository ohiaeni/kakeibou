## テスト方針

家計簿アプリケーション（MPA）の品質を保証するため、以下のテスト戦略を採用します：

1. **テストピラミッド**: 単体テスト > 統合テスト > E2Eテスト の順で重点を置く
2. **カバレッジ目標**: 80%以上の行カバレッジを維持
3. **TDD/BDD**: 新機能開発時はテストファーストアプローチを推奨
4. **継続的テスト**: CI/CDパイプラインでの自動テスト実行
5. **MPA特有のテスト**: フォーム処理、テンプレートレンダリング、セッション管理のテストを重視

## テスト分類

### 1. 単体テスト (Unit Tests)

#### 対象
- Service クラスのビジネスロジック
- Utility クラスの各メソッド
- ドメインモデルの振る舞い

#### テストフレームワーク
- **JUnit 5**: テスト実行フレームワーク
- **Spring Boot Test**: Spring コンテキストテスト
- **MockMvc**: MVCコントローラーテスト
- **Mockito**: モック作成
- **AssertJ**: アサーション
- **Testcontainers**: データベーステスト
- **@WithMockUser**: Spring Security認証テスト

#### サンプルテスト
```java
@ExtendWith(MockitoExtension.class)
class ExpenseServiceTest {

    @Mock
    private ExpenseRepository expenseRepository;

    @InjectMocks
    private ExpenseService expenseService;

    @Test
    @DisplayName("支出記録作成 - 正常ケース")
    void createExpense_Success() {
        // Given
        CreateExpenseRequest request = CreateExpenseRequest.builder()
            .categoryId(UUID.randomUUID())
            .amount(1500)
            .description("テスト支出")
            .expenseDate(LocalDate.now())
            .build();

        // When & Then
        assertThatNoException()
            .isThrownBy(() -> expenseService.createExpense(request));
    }

    @Test
    @DisplayName("支出記録作成 - 不正な金額でバリデーションエラー")
    void createExpense_InvalidAmount_ThrowsException() {
        // Given
        CreateExpenseRequest request = CreateExpenseRequest.builder()
            .amount(-100) // 不正な金額
            .build();

        // When & Then
        assertThatThrownBy(() -> expenseService.createExpense(request))
            .isInstanceOf(ValidationException.class)
            .hasMessageContaining("金額は1円以上である必要があります");
    }
}
```

### 2. 統合テスト (Integration Tests)

#### 対象
- Repository とデータベースの連携
- Spring Boot アプリケーションコンテキスト
- 外部システムとの連携

#### テスト構成
```java
@SpringBootTest
@TestPropertySource(properties = {
    "spring.datasource.url=jdbc:h2:mem:testdb",
    "spring.jpa.hibernate.ddl-auto=create-drop"
})
@Transactional
class ExpenseRepositoryIntegrationTest {

    @Autowired
    private ExpenseRepository expenseRepository;

    @Autowired
    private TestEntityManager entityManager;

    @Test
    @DisplayName("ユーザーIDで支出記録を検索")
    void findByUserId_ReturnsExpenses() {
        // Given
        UUID userId = UUID.randomUUID();
        ExpenseEntity expense = createTestExpense(userId);
        entityManager.persistAndFlush(expense);

        // When
        List<ExpenseEntity> results = expenseRepository.findByUserId(userId);

        // Then
        assertThat(results).hasSize(1);
        assertThat(results.get(0).getUserId()).isEqualTo(userId);
    }
}
```

### 3. コントローラーテスト (Controller Tests)

#### 対象
- MVCコントローラーのエンドポイント
- フォーム処理とバリデーション
- テンプレートレンダリング
- 認証・認可

#### テスト構成
```java
@SpringBootTest
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
@TestPropertySource(properties = {
    "spring.datasource.url=jdbc:h2:mem:testdb"
})
class ExpenseControllerTest {

    @Autowired
    private MockMvc mockMvc;

    @MockBean
    private ExpenseService expenseService;

    @Test
    @DisplayName("GET /expenses - 支出履歴画面表示")
    @WithMockUser(username = "testuser")
    void getExpenses_ReturnsExpenseListPage() throws Exception {
        // Given
        List<ExpenseDTO> expenses = Arrays.asList(
            ExpenseDTO.builder()
                .id(UUID.randomUUID())
                .amount(1500)
                .description("テスト支出")
                .build()
        );
        given(expenseService.getExpensesByUser(any())).willReturn(expenses);

        // When & Then
        mockMvc.perform(get("/expenses"))
            .andExpect(status().isOk())
            .andExpect(view().name("expenses/list"))
            .andExpect(model().attributeExists("expenses"))
            .andExpect(model().attribute("expenses", hasSize(1)));
    }

    @Test
    @DisplayName("POST /expenses - 支出記録作成")
    @WithMockUser(username = "testuser")
    void createExpense_ValidForm_RedirectsToExpenseList() throws Exception {
        // When & Then
        mockMvc.perform(post("/expenses")
                .with(csrf())
                .param("amount", "1500")
                .param("categoryId", UUID.randomUUID().toString())
                .param("description", "テスト支出")
                .param("expenseDate", LocalDate.now().toString()))
            .andExpect(status().is3xxRedirection())
            .andExpect(redirectedUrl("/expenses"));
    }

    @Test
    @DisplayName("POST /expenses - バリデーションエラー")
    @WithMockUser(username = "testuser")
    void createExpense_InvalidForm_ReturnsFormWithErrors() throws Exception {
        // When & Then
        mockMvc.perform(post("/expenses")
                .with(csrf())
                .param("amount", "-100") // 不正な金額
                .param("description", ""))
            .andExpect(status().isOk())
            .andExpected(view().name("expenses/form"))
            .andExpect(model().hasErrors());
    }
}
```

### 4. セキュリティテスト

#### 認証テスト
```java
@Test
@DisplayName("認証なしでページアクセス - ログイン画面にリダイレクト")
void accessPageWithoutAuth_RedirectsToLogin() throws Exception {
    // When & Then
    mockMvc.perform(get("/expenses"))
        .andExpect(status().is3xxRedirection())
        .andExpected(redirectedUrlPattern("**/login"));
}

@Test
@DisplayName("不正なセッション - ログイン画面にリダイレクト")
void accessPageWithInvalidSession_RedirectsToLogin() throws Exception {
    // Given
    MockHttpSession session = new MockHttpSession();
    session.invalidate();

    // When & Then
    mockMvc.perform(get("/expenses").session(session))
        .andExpect(status().is3xxRedirection())
        .andExpected(redirectedUrlPattern("**/login"));
}

@Test
@DisplayName("CSRF トークンなしでPOST - 403エラー")
void postWithoutCsrfToken_Returns403() throws Exception {
    // When & Then
    mockMvc.perform(post("/expenses")
            .param("amount", "1500"))
        .andExpect(status().isForbidden());
}
}
```

## テストのベストプラクティス

### 1. テスト命名規則
- テストクラス: `[対象クラス名]Test`
- テストメソッド: `[メソッド名]_[条件]_[期待結果]`
- 日本語での説明: `@DisplayName` アノテーションを活用

### 2. テストの構造 (AAA パターン)
```java
@Test
void testMethod() {
    // Arrange (Given) - テストデータの準備
    
    // Act (When) - テスト対象メソッドの実行
    
    // Assert (Then) - 結果の検証
}
```

### 3. テストの独立性
- 各テストは独立して実行可能であること
- テスト間でデータを共有しない
- `@Transactional` や `@DirtiesContext` を適切に使用

### 4. モックの適切な使用
- 外部依存をモック化
- 過度なモック化は避ける
- 実際の動作に近い形でテスト
