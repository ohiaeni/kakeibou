# テスト戦略

## テスト方針

家計簿アプリケーションの品質を保証するため、以下のテスト戦略を採用します：

1. **テストピラミッド**: 単体テスト > 統合テスト > E2Eテスト の順で重点を置く
2. **カバレッジ目標**: 80%以上の行カバレッジを維持
3. **TDD/BDD**: 新機能開発時はテストファーストアプローチを推奨
4. **継続的テスト**: CI/CDパイプラインでの自動テスト実行

## テスト分類

### 1. 単体テスト (Unit Tests)

#### 対象
- Service クラスのビジネスロジック
- Utility クラスの各メソッド
- ドメインモデルの振る舞い

#### テストフレームワーク
- **JUnit 5**: テスト実行フレームワーク
- **Mockito**: モック作成
- **AssertJ**: アサーション
- **Testcontainers**: データベーステスト

#### サンプルテスト
```java
@ExtendWith(MockitoExtension.class)
class ExpenseServiceTest {

    @Mock
    private ExpenseRepository expenseRepository;

    @InjectMocks
    private ExpenseService expenseService;

    @Test
    @DisplayName("支出記録作成 - 正常ケース")
    void createExpense_Success() {
        // Given
        CreateExpenseRequest request = CreateExpenseRequest.builder()
            .categoryId(UUID.randomUUID())
            .amount(1500)
            .description("テスト支出")
            .expenseDate(LocalDate.now())
            .build();

        // When & Then
        assertThatNoException()
            .isThrownBy(() -> expenseService.createExpense(request));
    }

    @Test
    @DisplayName("支出記録作成 - 不正な金額でバリデーションエラー")
    void createExpense_InvalidAmount_ThrowsException() {
        // Given
        CreateExpenseRequest request = CreateExpenseRequest.builder()
            .amount(-100) // 不正な金額
            .build();

        // When & Then
        assertThatThrownBy(() -> expenseService.createExpense(request))
            .isInstanceOf(ValidationException.class)
            .hasMessageContaining("金額は1円以上である必要があります");
    }
}
```

### 2. 統合テスト (Integration Tests)

#### 対象
- Repository とデータベースの連携
- Spring Boot アプリケーションコンテキスト
- 外部システムとの連携

#### テスト構成
```java
@SpringBootTest
@TestPropertySource(properties = {
    "spring.datasource.url=jdbc:h2:mem:testdb",
    "spring.jpa.hibernate.ddl-auto=create-drop"
})
@Transactional
class ExpenseRepositoryIntegrationTest {

    @Autowired
    private ExpenseRepository expenseRepository;

    @Autowired
    private TestEntityManager entityManager;

    @Test
    @DisplayName("ユーザーIDで支出記録を検索")
    void findByUserId_ReturnsExpenses() {
        // Given
        UUID userId = UUID.randomUUID();
        ExpenseEntity expense = createTestExpense(userId);
        entityManager.persistAndFlush(expense);

        // When
        List<ExpenseEntity> results = expenseRepository.findByUserId(userId);

        // Then
        assertThat(results).hasSize(1);
        assertThat(results.get(0).getUserId()).isEqualTo(userId);
    }
}
```

### 3. API テスト (API Tests)

#### 対象
- REST API エンドポイント
- 認証・認可
- リクエスト/レスポンスの形式

#### テスト構成
```java
@SpringBootTest(webEnvironment = SpringBootTest.WebEnvironment.RANDOM_PORT)
@AutoConfigureTestDatabase(replace = AutoConfigureTestDatabase.Replace.NONE)
class ExpenseControllerApiTest {

    @Autowired
    private TestRestTemplate restTemplate;

    @Test
    @DisplayName("GET /api/v1/expenses - 支出記録一覧取得")
    void getExpenses_ReturnsExpenseList() {
        // Given
        HttpHeaders headers = createAuthHeaders();
        HttpEntity<String> entity = new HttpEntity<>(headers);

        // When
        ResponseEntity<ExpenseListResponse> response = restTemplate.exchange(
            "/api/v1/expenses",
            HttpMethod.GET,
            entity,
            ExpenseListResponse.class
        );

        // Then
        assertThat(response.getStatusCode()).isEqualTo(HttpStatus.OK);
        assertThat(response.getBody().getExpenses()).isNotEmpty();
    }
}
```

### 4. セキュリティテスト

#### 認証テスト
```java
@Test
@DisplayName("認証なしでAPIアクセス - 401エラー")
void accessApiWithoutAuth_Returns401() {
    // When
    ResponseEntity<String> response = restTemplate.getForEntity(
        "/api/v1/expenses", String.class);

    // Then
    assertThat(response.getStatusCode()).isEqualTo(HttpStatus.UNAUTHORIZED);
}

@Test
@DisplayName("不正なJWTトークン - 401エラー")
void accessApiWithInvalidToken_Returns401() {
    // Given
    HttpHeaders headers = new HttpHeaders();
    headers.setBearerAuth("invalid-token");
    HttpEntity<String> entity = new HttpEntity<>(headers);

    // When
    ResponseEntity<String> response = restTemplate.exchange(
        "/api/v1/expenses", HttpMethod.GET, entity, String.class);

    // Then
    assertThat(response.getStatusCode()).isEqualTo(HttpStatus.UNAUTHORIZED);
}
```

## パフォーマンステスト

### 1. 負荷テスト
JMeterまたはGatlingを使用して以下をテスト：
- 同時ユーザー数: 100人
- 継続時間: 10分間
- 対象エンドポイント: `/api/v1/expenses`

### 2. レスポンス時間目標
- API レスポンス: 95%のリクエストが200ms以内
- データベースクエリ: 95%のクエリが50ms以内

## テスト実行とレポート

### 1. テスト実行コマンド

```bash
# 全テスト実行
./gradlew test

# 特定のテストクラス実行
./gradlew test --tests ExpenseServiceTest

# 特定のテストメソッド実行
./gradlew test --tests ExpenseServiceTest.createExpense_Success

# タグベースでテスト実行
./gradlew test -Dtags="integration"
```

### 2. カバレッジレポート

```bash
# カバレッジレポート生成
./gradlew jacocoTestReport

# カバレッジ閾値チェック
./gradlew jacocoTestCoverageVerification
```

### 3. CI/CD での自動テスト

```yaml
# .github/workflows/test.yml の例
name: Tests
on: [push, pull_request]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v2
      - uses: actions/setup-java@v2
        with:
          java-version: '17'
      - name: Run tests
        run: ./gradlew test jacocoTestReport
      - name: Upload coverage
        uses: codecov/codecov-action@v1
```

## テストのベストプラクティス

### 1. テスト命名規則
- テストクラス: `[対象クラス名]Test`
- テストメソッド: `[メソッド名]_[条件]_[期待結果]`
- 日本語での説明: `@DisplayName` アノテーションを活用

### 2. テストの構造 (AAA パターン)
```java
@Test
void testMethod() {
    // Arrange (Given) - テストデータの準備
    
    // Act (When) - テスト対象メソッドの実行
    
    // Assert (Then) - 結果の検証
}
```

### 3. テストの独立性
- 各テストは独立して実行可能であること
- テスト間でデータを共有しない
- `@Transactional` や `@DirtiesContext` を適切に使用

### 4. モックの適切な使用
- 外部依存をモック化
- 過度なモック化は避ける
- 実際の動作に近い形でテスト
