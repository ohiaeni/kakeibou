# アーキテクチャ設計

## システム概要

家計簿アプリは、ユーザーが日々の収支を記録・管理するためのWebアプリケーションです。Spring Bootを基盤としたマルチモジュール構成で構築されており、クリーンアーキテクチャの原則に従って設計されています。

## アーキテクチャ図

```
┌─────────────────┐    ┌─────────────────┐    ┌─────────────────┐
│      Web        │    │ Application     │    │ Infrastructure  │
│    (Controller) │───▶│     Core        │───▶│   (Repository)  │
│                 │    │   (Service)     │    │                 │
└─────────────────┘    └─────────────────┘    └─────────────────┘
                                │
                                ▼
                       ┌─────────────────┐
                       │ System Common   │
                       │   (Utilities)   │
                       └─────────────────┘
```

## モジュール構成

### 1. Web層 (web)
- **責務**: HTTP リクエストの受信とレスポンスの返却
- **主要コンポーネント**:
  - REST Controller
  - DTO (Data Transfer Object)
  - 例外ハンドラー
  - セキュリティ設定

### 2. アプリケーション層 (application-core)
- **責務**: ビジネスロジックの実装
- **主要コンポーネント**:
  - Service クラス
  - ドメインモデル
  - ユースケース実装

### 3. インフラストラクチャ層 (infrastructure)
- **責務**: 外部システムとの連携（データベース、外部API等）
- **主要コンポーネント**:
  - Repository 実装
  - MyBatis Mapper
  - エンティティクラス
  - データベース設定

### 4. システム共通層 (system-common)
- **責務**: 全モジュールで共通利用される機能
- **主要コンポーネント**:
  - ユーティリティクラス
  - 共通例外クラス
  - 定数定義

## 技術スタック

### バックエンド
- **フレームワーク**: Spring Boot 3.5.4
- **言語**: Java
- **ORM**: MyBatis 3.0.4
- **データベース**: PostgreSQL (本番環境), H2 (開発・テスト環境)
- **セキュリティ**: Spring Security
- **API仕様書**: SpringDoc OpenAPI 2.8.9

### 開発・運用ツール
- **ビルドツール**: Gradle
- **コード品質**: SpotBugs, Checkstyle, Spotless
- **テストカバレッジ**: JaCoCo
- **ログ**: SLF4J + Logback

## データベース設計原則

1. **UUID主キー**: 全テーブルでUUIDを主キーとして使用
2. **外部キー制約**: データ整合性を保証するため適切な外部キー制約を設定
3. **論理削除**: 重要なデータについては物理削除ではなく論理削除を採用
4. **監査ログ**: 作成日時、更新日時、作成者、更新者の記録

## セキュリティ考慮事項

1. **認証・認可**: Spring Securityによるロールベースアクセス制御
2. **パスワード**: BCryptによるハッシュ化
3. **SQL インジェクション対策**: MyBatisのパラメータバインディング使用
4. **CSRF対策**: Spring SecurityのCSRF保護機能を有効化

## パフォーマンス考慮事項

1. **データベース接続**: コネクションプールの適切な設定
2. **クエリ最適化**: 適切なインデックスの設定とN+1問題の回避
3. **キャッシュ**: Spring Cacheによる適切なキャッシュ戦略
4. **非同期処理**: 重い処理については非同期実行を検討
